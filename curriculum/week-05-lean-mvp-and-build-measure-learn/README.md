# Week 5 — Lean MVP & Build-Measure-Learn

Now you build — but not the product. You build the smallest possible *experiment* that tests your single riskiest assumption. The most common first-time mistake is treating the MVP as "version one of the real thing, just smaller." It isn't. An MVP is the cheapest way to learn whether your validated problem can be solved in a way people will adopt. Sometimes that is a landing page; sometimes it is you doing the work by hand behind a curtain. The lean **build-measure-learn** loop turns each MVP into a decision: persevere or pivot.

The deliverable is a built MVP (any fidelity) wired to one clear metric, plus the result and your persevere/pivot decision.

## Learning objectives

By the end of this week, you will be able to:

- **Identify** your single **riskiest assumption** and the cheapest test of it.
- **Choose an MVP type** that fits the assumption: smoke test / fake door, concierge, Wizard of Oz, or clickable prototype.
- **Define one metric** that the MVP will move (or not), and a threshold decided in advance.
- **Run the build-measure-learn loop** and read the result honestly.
- **Decide** persevere or pivot, and write down why.

## Topics covered

- MVP as experiment, not tiny product
- Riskiest-assumption thinking: what must be true for this to work?
- MVP types and when to use each
- Build → measure → learn, and the validated-learning idea
- Picking a single success metric and a pre-committed threshold
- Persevere vs. pivot, honestly

## In-class activities

- **Riskiest-assumption surfacing.** Each founder lists the assumptions their idea depends on, then ranks by "if this is false, the whole thing dies." The top one is the MVP target.
- **MVP-type matching.** Given an assumption, the group picks the cheapest MVP type that could test it. Most can be tested without building software.
- **Threshold pre-commit.** Each founder writes the number that would count as success *before* launching, so they can't move the goalposts later.

## Worked example

The clinic founder's riskiest assumption is "front desks will adopt a waitlist auto-fill tool." The cheapest test is not building it — it is a **concierge MVP**: for two clinics, she manually texts waitlisted patients when a cancellation comes in, tracking how many slots she refills. Success threshold, set in advance: refill at least 30% of cancellations across a week. She refills 45%. The assumption holds; she perseveres to a simple software version. Had she refilled 5%, she would pivot — perhaps the bottleneck was patient response time, not the front desk. Either way she learned in a week for almost no money.
